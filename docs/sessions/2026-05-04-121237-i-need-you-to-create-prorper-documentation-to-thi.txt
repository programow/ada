‚ĚĮ what is the porpuse of C and B? and use cases?                                
‚Źļ Good question. Both solve the same core problem: running tasks across many 
  packages efficiently. They differ in how heavyweight they are.

  Turborepo (B)

  What it does: A task runner that sits on top of your package manager. You
  define a "pipeline" (e.g., build depends on ^build of dependencies). Then
  turbo run test figures out which packages are affected, runs them in parallel
  where it can, and caches outputs keyed by file hashes ‚ÄĒ re-running an
  unchanged package is instant.

  Use cases:
  - 5+ packages where running bun test on each one serially takes minutes
  - CI where you want to skip rebuilding/retesting packages that didn't change
  (huge time saver on PRs)
  - Shared libraries: when packages/ui changes, automatically retest every app
  that depends on it
  - Remote cache shared across team/CI (e.g. Vercel-hosted) ‚ÄĒ your teammate's
  build result becomes your cache hit

  When it's overkill: Repos with 2‚Äď3 packages that don't share much code. The
  turbo.json config + cache plumbing costs more than it saves.

  Nx (C)

  What it does: Everything Turbo does, plus a project graph Nx maintains for
  you, plus code generators (nx generate @nx/react:lib my-lib scaffolds a new
  package wired into the workspace), plus opinionated plugins per framework
  (Next.js, React, Node, etc.).

  Use cases:
  - Large enterprise monorepos (50+ packages, multiple teams) where you
  genuinely need code generation, dependency graph visualization, enforced
  module boundaries (apps/web is forbidden from importing apps/admin)
  - Teams that want a "batteries included" framework rather than assembling
  tools themselves
  - Heavy use of generators ‚ÄĒ e.g., creating a new microservice from a template

  When it's overkill: Anything smaller than ~10 packages, or solo/small-team
  projects. Nx imposes its own conventions and adds a learning curve.

  For your repo

  You'll have 2 packages (desktop, landing) that share almost nothing (one is
  Tauri/Rust+TS, the other is Next.js). They don't form a build dependency
  graph. Caching gives you near-zero benefit because each tool already has its
  own cache (Tauri/Cargo, Next.js, Vite). My recommendation stays at A (Bun
  workspaces only).

  Pick: A, B, or C?

  Q8: Auto-update ‚ÄĒ pick the manifest hosting strategy.

  Tauri ships an official updater plugin (@tauri-apps/plugin-updater + Rust
  side). The flow is: app polls a JSON manifest ‚Üí manifest lists latest version
  + signed download URLs ‚Üí app downloads the platform-appropriate installer ‚Üí
  verifies a minisign signature against an embedded public key ‚Üí installs. The
  signing key is generated once, public key bakes into the app, private key
  lives as a CI secret.

  Your release artifacts go to GitHub Releases. The question is where the 
  latest.json manifest lives so the app can poll a stable URL:

  - A) S3 bucket (same one as the landing page). CI uploads latest.json to
  https://yoursite/updates/latest.json after each release. App polls that URL.
  Pro: one infra, no rate limits, full control over caching/CDN. Con: one extra
  CI step per release.
  - B) GitHub Releases asset (latest.json attached to a "latest" release). App
  polls https://github.com/programow/ada/releases/latest/download/latest.json.
  Pro: zero extra infra ‚ÄĒ CI just uploads alongside the binaries. Con: GitHub
  anonymous rate limits (60/hour/IP) can bite if many users open the app
  simultaneously, and you can't put the manifest behind a CDN you control.
  - C) GitHub Releases via the API (no static manifest). Tauri's updater can

‚Źļ Q15: Code signing & Linux distribution format

  Honest reality check before the question: code signing is the difference 
  between "downloads and runs" vs. "scary OS warning that turns 80% of users 
  away." This is a real cost (money + ops), and the choice differs per platform.

  The three platforms:

  macOS ‚ÄĒ Distribution outside the App Store requires:
  - Apple Developer Program ($99/year)
  - Developer ID Application certificate (sign the binary)
  - Notarization (Apple scans your binary, returns a ticket, you staple it to
  the DMG)
  - Result: Clean install. No warnings. Required if you want to distribute
  outside the App Store at all ‚ÄĒ unsigned macOS apps now get aggressive
  Gatekeeper blocks.

  Windows ‚ÄĒ Microsoft SmartScreen flags unsigned executables as "unrecognized
  publisher." Two cert tiers:
  - OV (Organization Validation) cert ‚ÄĒ ~$200‚Äď400/year from a CA. Software-based
   key. Reduces warnings, builds SmartScreen reputation slowly with downloads.
  - EV (Extended Validation) cert ‚ÄĒ ~$300‚Äď600/year. Hardware token (USB key
  shipped to you). Instant SmartScreen reputation. Required for Windows kernel
  drivers; overkill for a desktop app.
  - Unsigned ‚ÄĒ every Windows download triggers SmartScreen "unrecognized
  publisher" warning; user clicks through "More info ‚Üí Run anyway." Free but bad
   first-run UX.

  Linux ‚ÄĒ Three reasonable formats:
  - AppImage ‚ÄĒ single executable, no install, no signing required by convention.
   Tauri builds this by default. Easiest. Users chmod +x and run.
  - deb / rpm ‚ÄĒ system packages, can be GPG-signed (free, you generate the key),
   shows up in apt/dnf. More "installed" feel but more pieces to maintain.
  - Flatpak / Snap ‚ÄĒ sandboxed, distributed via Flathub or Snap Store. More
  setup, but discoverability via app stores. Defer.

  Honest recommendation for a pre-release BYOK app with zero users:

  - macOS: sign + notarize. $99/year is the cost of distributing a real Mac app.
   Skip this and the install ritual stops being "download + drag" and becomes
  "right-click + Open + System Settings + Allow." Don't ship Mac without it.
  - Windows: ship unsigned at v1, sign once download counts justify it.
  SmartScreen reputation has to build with downloads anyway ‚ÄĒ paying $300/year
  before you have users is wasted because the cert won't have reputation yet
  either. Document the warning prominently in install instructions. Add OV cert
  in v2 when you have measurable download volume.
  - Linux: AppImage only at v1. No signing concern, single artifact, works
  everywhere. Add deb/rpm in v2 if there's demand.

  Q15 answers needed:
  1. macOS signing ‚ÄĒ confirm Apple Developer Program ($99/yr), or skip Mac 
  entirely at v1? I'd say sign.
  2. Windows signing ‚ÄĒ unsigned at v1 (recommended), OV cert at v1, or EV cert 
  at v1? I'd say unsigned with prominent docs.
  3. Linux format ‚ÄĒ AppImage only at v1 (recommended), or also deb/rpm? I'd say
  AppImage.

  If "follow your recommendations" works, just say so.

  What "available in package managers" actually means

  - Setup: ~30 minutes. Ongoing: zero (CI does it).

  - Setup: ~half a day to a full day. Ongoing: the CI scripts need occasional

  - Result: users get apt install ada and apt upgrade for free.

  Q17: Where does audio capture happen ‚ÄĒ webview (MediaRecorder) or Rust (cpal)?

  This is genuinely architecturally important. There's a real choice with real
  trade-offs.

  A) Webview captures via MediaRecorder + getUserMedia (the current Electron
  approach, ported to Tauri).
  - Pro: Less Rust code; permission UX is browser-handled; well-known API.
  - Con: Format support varies per platform. Tauri uses native webviews ‚ÄĒ
  WKWebView (macOS), WebView2 (Windows), WebKitGTK (Linux). MediaRecorder
  support in WKWebView has historical quirks (mp4 only in newer macOS, no webm),
   WebView2 supports webm, WebKitGTK supports Opus/Vorbis. You end up writing
  format-negotiation code (MediaRecorder.isTypeSupported() switching) and the
  audio fixtures vary per platform ‚ÄĒ testing gets gnarly.
  - Con: Cross-platform consistency is hard to guarantee; subtle bugs surface
  only on the platform you're not developing on.

  B) Rust captures via cpal (cross-platform audio I/O crate).
  - Pro: Identical code path on all 3 platforms ‚ÄĒ cpal handles CoreAudio
  (macOS), WASAPI (Windows), ALSA/PulseAudio (Linux) uniformly. Output is
  consistent (raw PCM ‚Üí wav).
  - Pro: Audio capture is testable ‚ÄĒ fixtures are wav files in the same format,
  mock cpal in unit tests, real cpal in integration tests. No platform-specific
  gotchas.
  - Pro: Future loopback support has a natural home ‚ÄĒ same AudioSource trait,
  just add a LoopbackSource impl using ScreenCaptureKit / WASAPI loopback /
  PipeWire. The webview doesn't change.
  - Pro: Permission requests (microphone) handled by Rust ‚Üí cleaner integration
  with Tauri's permission model.
  - Con: More Rust code (~150‚Äď200 lines vs. ~30 lines of TS). Microphone
  permission UX is now your responsibility (Rust requests via system dialog, you
   wire it up).

  C) Hybrid ‚ÄĒ webview decides "start" / "stop", Rust captures and stores the
  buffer, returns it via a Tauri command.
  - Pro: UI is webview-driven, audio is Rust-driven (best of both).
  - This is what option B actually means in practice. The webview's role is just
   "tell Rust to start/stop and show recording state." Pure UI, no audio bytes
  in TS.

  My strong recommendation: B (which is effectively C). Cross-platform
  consistency, testability, and the future-loopback-without-rewrite story all
  point the same direction. The MediaRecorder approach was right for the
  original Mac-only Electron app; for cross-platform Tauri it's the wrong
  primitive. The extra Rust is bounded ‚ÄĒ cpal is a stable, well-documented
  crate, and the audio module is one of the cleanest places to put a trait-based
   seam for testing.

  Trade-off summary: A = less Rust, more platform-specific bugs. B = more Rust,
  fewer platform-specific bugs, much better testability and future-feature
  story.

  Pick A or B. My rec: B.

  Audio capture & permissions module

  Rust-side architecture (single audio module in src-tauri/):

  // packages/desktop/src-tauri/src/audio/mod.rs
  pub trait AudioSource: Send + Sync {
      fn check_permission(&self) -> PermissionState;
      fn request_permission(&self) -> Result<PermissionState>;
      fn start_capture(&self) -> Result<CaptureHandle>;
  }

  pub enum PermissionState {
      Granted,
      Denied,
      NotDetermined,
  }

  // Real implementation
  pub struct MicrophoneSource { /* cpal, platform-specific permission code */ }

  // Test implementation
  pub struct MockMicrophoneSource { /* loads audio fixtures, no OS calls */ }

  // Future implementation (v2)
  // pub struct LoopbackSource { /* ScreenCaptureKit / WASAPI / PipeWire */ }

  The trait pattern means tests use MockMicrophoneSource (loads
  tests/fixtures/audio/hello-world.wav), production uses MicrophoneSource.
  Adding loopback later = one new struct that implements the same trait.

  Permission flow per platform ‚ÄĒ the standard, working way

  macOS ‚ÄĒ the path that actually works:

  The TCC permission system is brittle in three specific ways, all of which we
  sidestep:

  1. TCC ties permissions to code signature. If the signature changes between
  builds (ad-hoc ‚Üí Developer ID, or different Developer IDs), permissions reset
  silently. Our solution: we sign with a stable Apple Developer ID across all
  releases. CI uses the same cert, signature stays consistent, TCC keeps the
  grant.
  2. Info.plist must declare NSMicrophoneUsageDescription. Without it, the OS
  terminates the app on first mic access with no UI feedback. Tauri configures
  this via tauri.conf.json:
  "macOS": {
    "infoPlist": {
      "NSMicrophoneUsageDescription": "Ada needs microphone access to transcribe
   your speech."
    }
  }
  3. Permission must be requested explicitly, not implicitly. The cleanest path
  is to call AVCaptureDevice.requestAccess(for: .audio) via a small Rust wrapper
   using objc2 crate, then open the mic via cpal. This triggers the OS prompt at
   a moment we control (e.g., during onboarding or before first recording).

  The Rust impl:
  #[cfg(target_os = "macos")]
  fn request_permission(&self) -> Result<PermissionState> {
      // objc2 call to AVCaptureDevice.requestAccess
      // returns Granted / Denied / NotDetermined synchronously after user 
  responds
  }

  Entitlements file (signed into the app):
  <key>com.apple.security.device.audio-input</key>
  <true/>

  When denied: UI shows a clear message + deep link button:
  "Microphone access denied. Open System Settings ‚Üí Privacy & Security ‚Üí
  Microphone, then enable Ada."
  [Open Settings] // opens
  x-apple.systempreferences:com.apple.preference.security?Privacy_Microphone

  This is the same pattern Wispr Flow, Whisper, and any well-built Mac dictation
   app uses. No tccutil reset rituals needed at runtime ‚ÄĒ that was a dev 
  workflow concern (when iterating on signatures), not a user concern. Once the
  cert is stable, TCC just works.

  Windows:

  - WASAPI has no per-app permission prompt; the system-wide Settings ‚Üí Privacy 
  ‚Üí Microphone toggle controls all apps.
  - We check the toggle state via Windows registry on app start
  (HKEY_CURRENT_USER\Software\Microsoft\Windows\CurrentVersion\CapabilityAccessM
  anager\ConsentStore\microphone) and surface a deep link if disabled:
  "Microphone is disabled in Windows Privacy settings."

  - xdg-desktop-portal (Flatpak/Snap context) does have a permission prompt, but
   we're not in those sandboxes at v1.

  Permission state UX (cross-platform)

  The desktop app's main UI shows current permission state at all times. Three
  states:

  - Granted ‚Üí recording works; show normal recording UI.
  - NotDetermined ‚Üí onboarding-style prompt: "Ada needs microphone access to

  - Stable signing cert removes the #1 cause of "my mic suddenly stopped
  working" ‚ÄĒ tccutil resets, sig changes, ad-hoc rebuild surprises. We avoid
  this entire class by being a real signed app.

  - Deep links to settings mean users don't navigate menus blind when
  something's denied ‚ÄĒ they tap and they're there.
  - Trait-based seam in Rust means we can write a permission-state state machine
   and unit-test it without OS calls.
  - AVCaptureDevice.requestAccess is Apple's documented happy path; everything
  else (Tauri plugins, half-documented workarounds) is a worse version of this.

  Are Windows Credential Manager and Linux Secret Service safe? Honest answer.

  Your audience is developers ‚ÄĒ they will read the privacy page and notice if we
   hand-wave. So the honest, complete answer:

  macOS Keychain (via keyring crate):
  - Hardware-backed encryption on T2/Apple Silicon Macs, software-backed on
  older Intel Macs
  - Per-app ACL: only the signed app (Ada specifically) can read its own secrets
   ‚ÄĒ other apps see "access denied" prompts
  - This is unambiguously the safe answer. Industry standard.

  Windows Credential Manager (via keyring crate, uses DPAPI under the hood):
  - Encrypts at rest with keys derived from the user's login credentials (Data
  Protection API)
  - Honest caveat: Credential Manager is per-user, not per-app. Any process
  running as the same user can read the credentials if it knows the schema. This
   is a known Windows architectural limitation, not a flaw in our
  implementation.
  - This is what 1Password (Windows agent), GitHub CLI, Azure CLI, Docker
  Desktop, AWS CLI, and roughly every developer tool uses on Windows. It's the
  platform standard.
  - Stronger options exist but they trade UX for marginal security:
    - DPAPI directly with explicit ACLs ‚Üí not exposed by keyring, custom code
    - Windows Hello-protected credentials ‚Üí requires biometric per access
  (terrible for an STT app where keys are read once per recording)
  - Verdict: standard, safe enough for the threat model (a malicious app already
   running as the user can do worse things than read your STT API key). Document
   this honestly.

  Linux Secret Service (via keyring crate, secret-service feature):
  - D-Bus API spec: org.freedesktop.secrets. Implementations: GNOME Keyring
  (most common) or KWallet (KDE).
  - Encrypts at rest with the user's session password.
  - Honest caveats:
    a. Per-user, not per-app (same as Windows ‚ÄĒ D-Bus secrets aren't isolated by
   app).
    b. Not all Linux installs have a Secret Service running. Headless servers,
  minimal i3/Sway setups, or distros where the user removed gnome-keyring will
  fail. The keyring crate returns a clean error in this case ‚ÄĒ we surface it
  with a helpful message ("This system has no secret service. Install
  gnome-keyring or libsecret-1-0 and re-run Ada.").
  - Verdict: standard, safe by Linux conventions, with a known availability 
  caveat. Document the prerequisite.
